## What is the Saga Pattern?

The **Saga Pattern** manages distributed transactions across multiple services by breaking them into a sequence of local transactions. Each step has a compensating transaction to undo its work if a later step fails.

---

## The Problem

Distributed transactions are hard:

```mermaid
flowchart LR
    O["Order Service"] --> P["Payment Service"] --> I["Inventory Service"]
    P -->|"Payment fails?"| X["Can't use 2PC easily\nNeed to rollback!"]
```

---

## How Sagas Work

```mermaid
flowchart LR
    subgraph "Happy Path"
        T1["T1"] --> T2["T2"] --> T3["T3"] --> T4["T4"] --> S["Success!"]
    end
    subgraph "Failure at T3"
        T1b["T1"] --> T2b["T2"] --> T3b["T3 (fails)"]
        T3b --> C2["C2"] --> C1["C1 (compensate backwards)"]
    end
```

| **Concept** | **Description** |
|------------|-----------------|
| Transaction (T) | Local transaction in a service |
| Compensation (C) | Undo action for a transaction |
| Saga | Sequence of T₁, T₂, ..., Tₙ with compensations C₁, C₂, ..., Cₙ |

---

## Saga Types

### Choreography

Services react to events, no central coordinator:

```mermaid
flowchart LR
    OS["Order\nService"] -->|"event"| PS["Payment\nService"] -->|"event"| IS["Inventory\nService"]
    IS -.->|"compensation"| PS -.->|"compensation"| OS
```

**Pros**: Decoupled, simple for few steps
**Cons**: Hard to track, complex flows

### Orchestration

Central coordinator manages the flow:

```mermaid
flowchart TD
    ORC["Orchestrator"] --> OS["Order Service"]
    ORC --> PS["Payment Service"]
    ORC --> IS["Inventory Service"]
```

**Pros**: Easy to understand, centralized logic
**Cons**: Orchestrator can be bottleneck

## Example: Order Saga

```mermaid
flowchart TD
    S1["1. Create Order (Pending)\nCompensation: Cancel Order"] --> S2["2. Reserve Inventory\nCompensation: Release Inventory"]
    S2 --> S3["3. Process Payment\nCompensation: Refund Payment"]
    S3 --> S4["4. Confirm Order\n(final step)"]
```

### Failure Scenario

```mermaid
flowchart TD
    A["1. Create Order ✓"] --> B["2. Reserve Inventory ✓"]
    B --> C["3. Process Payment ✗"]
    C -->|"Compensate"| D["Release Inventory"]
    D --> E["Cancel Order"]
```

## Implementation Tips

| **Consideration** | **Approach** |
|------------------|-------------|
| Idempotency | Make operations repeatable safely |
| Correlation ID | Track saga instance across services |
| Timeout | Handle stuck sagas |
| Monitoring | Track saga state and failures |

---

## Saga vs 2PC

| **Aspect** | **Saga** | **2PC** |
|-----------|---------|---------|
| Consistency | Eventual | Strong |
| Availability | Higher | Lower |
| Complexity | Business logic | Protocol |
| Scalability | Better | Worse |
| Recovery | Compensation | Rollback |
